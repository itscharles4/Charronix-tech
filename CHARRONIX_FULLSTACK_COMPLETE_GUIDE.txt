 1. PROJECT OVERVIEW & VISION
================================================================================

Charronix School Management System is a full-stack web application designed to
digitize and streamline school operations. It provides role-specific portals for
Administrators, Principals, Teachers, Students, and Parents.

Current Status (February 2026):
  - Frontend: COMPLETE (React + TypeScript + Vite, 11 components)
  - Backend:  COMPLETE (Express.js + TypeScript, 19 API modules)
  - Database: COMPLETE (PostgreSQL 17, 31 tables, migrated + seeded)
  - Cache:    COMPLETE (Redis 5.x, enabled with 100MB limit)
  - Auth:     COMPLETE (JWT Access + Refresh tokens, RBAC)

Roles Supported:
  ADMIN       - Full system control
  PRINCIPAL   - School-wide view and management
  TEACHER     - Class and student management
  STUDENT     - Self-view portal (grades, attendance, notices)
  PARENT      - Child progress view portal

--- 4.3 LOGIN SYSTEM & ID FORMAT RULES ---

  Each role uses a different ID format validated by regex:

  Role       | ID Format                      | Example          | Regex
  -----------|--------------------------------|------------------|----------------------
  Student    | 2 digits + 3 letters + 4 digits | 24BIT0522       | ^\d{2}[A-Za-z]{3}\d{4}$
  Teacher    | Exactly 6 digits               | 100001           | ^\d{6}$
  Principal  | Exactly 6 digits               | 900001           | ^\d{6}$
  Parent     | Student first name + DDMMYYYY  | Aarav15032008    | ^[A-Za-z]+(0[1-9]|...)$

  Login Flow:
  1. User selects portal role
  2. Enters their custom ID + password
  3. handleLoginSubmit() maps ID → email, calls backend API
  4. Backend returns JWT accessToken + refreshToken
  5. Tokens stored in localStorage
  6. User redirected to their dashboard

--- 5.1 ARCHITECTURE PATTERN ---

  Pattern: MVC (Model-View-Controller) + Service Layer

  Client (React)
     ↓  HTTP Request
  Express Router
     ↓
  Middleware (Rate Limit → CORS → Helmet → Auth → Role → Validate)
     ↓
  Controller (handles request, calls service)
     ↓
  Service Layer (business logic, cache check/set)
     ↓
  Prisma ORM (type-safe database queries)
     ↓
  PostgreSQL (data storage)
     ↓  Response
  ApiResponse utility (standardized JSON format)
     ↓
  Client

--- 5.2 REQUEST PIPELINE (MIDDLEWARE STACK) ---

  Order  | Middleware         | What it does
  -------|--------------------|-------------------------------------------------
  1      | Rate Limiter       | 100 requests per 15 minutes per IP
  2      | CORS               | Allow: localhost:3000, localhost:5173
  3      | Helmet             | Set security HTTP headers
  4      | express.json()     | Parse JSON request bodies
  5      | Morgan             | Log all HTTP requests to console
  6      | Route Handler      | Match URL to controller
  7      | Zod Validate       | Validate request body/params/query
  8      | authMiddleware     | Verify JWT access token
  9      | roleMiddleware     | Check user role has permission
  10     | Controller         | Execute business logic

  Error Handler (catch-all at end, returns structured error response)

--- 7.2 CACHE STRATEGY ---

  What is cached:
  ─────────────────────────────────────────────────────
  Dashboard KPIs          TTL: 1 hour
  School settings         TTL: 6 hours
  Active timetable        TTL: 1 hour
  Notice lists            TTL: 30 minutes
  Rate limit counters     TTL: 15 minutes (per IP)
  Blacklisted JWT tokens  TTL: Until token expiry

  Key naming convention: resource:id:field
  Example: student:uuid:grades, dashboard:stats

--- 7.3 CACHE CONFIGURATION IN CODE ---

  File: backend/src/services/cache.service.ts
  Methods:
    cache.get<T>(key)             → T | null
    cache.set(key, value, ttl)    → void
    cache.delete(key)             → void
    cache.deletePattern(pattern)  → void (bulk invalidation)
    cache.exists(key)             → boolean
    cache.increment(key, ttl)     → number (for rate limiting)
    cache.getClient()             → Redis instance

  Connection is wrapped in retryStrategy (exponential backoff).
  If Redis is unavailable, all cache methods return null/false gracefully.

 8. AUTHENTICATION & AUTHORIZATION
================================================================================

--- 8.1 JWT TOKEN FLOW ---

  1. User logs in → POST /api/v1/auth/login { email, password }
  2. Backend verifies password with bcrypt
  3. Backend generates:
       accessToken  (JWT, 15 minute expiry)
       refreshToken (JWT, 7 day expiry)
  4. Frontend stores both in localStorage
  5. All protected requests include: Authorization: Bearer <accessToken>
  6. When access token expires → POST /auth/refresh with refreshToken
  7. Logout → refreshToken blacklisted in Redis

  JWT Payload:
    { userId, email, role, iat, exp }

--- 8.2 ROLE-BASED ACCESS CONTROL (RBAC) ---

13. DEPLOYMENT GUIDE
================================================================================

  Option A: Local (Current — Development)
    → Run both npm run dev servers (see Section 10)

  Option B: Railway / Render (Easiest Cloud)
    1. Push code to GitHub
    2. Connect repo to Railway.app or Render.com
    3. Add PostgreSQL + Redis as services
    4. Set all environment variables
    5. Auto-deploys on git push to main branch

  Option C: Docker (Recommended for Production)

    docker-compose.yml:
      version: '3.8'
      services:
        db:
          image: postgres:17
          environment:
            POSTGRES_DB: charronix_db
            POSTGRES_USER: charronix_user
            POSTGRES_PASSWORD: charronix_pass_2024
          ports:
            - "5432:5432"
          volumes:
            - pgdata:/var/lib/postgresql/data

        redis:
          image: redis:7
          ports:
            - "6379:6379"
          command: redis-server --maxmemory 100mb

        backend:
          build: ./backend
          ports:
            - "5000:5000"
          depends_on:
            - db
            - redis
          env_file: ./backend/.env

        frontend:
          build: .
          ports:
            - "3000:3000"
          depends_on:
            - backend

      volumes:
        pgdata:

    Run:
      docker-compose up -d

  Option D: VPS (DigitalOcean / AWS EC2)
    1. Install Node.js 20, PostgreSQL 17, Redis, Nginx
    2. Clone repo on server
    3. Run backend with PM2: pm2 start dist/server.js
    4. Configure Nginx as reverse proxy on port 80/443
    5. Set up SSL with Let's Encrypt (certbot)
    6. Build frontend: npm run build → serve with Nginx
